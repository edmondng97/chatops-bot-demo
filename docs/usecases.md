# 业务功能清单（Use Cases）

- **来源**：当前代码工作区 `/Users/edmond/Desktop/chatops-bot-demo`（分支 `feat/demo-build`），逐个对外入口核对。
- **用途**：作为业务功能基线，供 QA / 运营 / PM 核对功能是否齐全、迁移或重构后有无遗漏。
- **面向对象**：QA 与非技术同学。
- **原则**：只描述运营 / QA 可观察到的**业务语义与业务效果**，不含任何技术实现细节（不写调用链、时序、数据结构、类名/方法名）。
- **图例**：
  - ✅ 主干正常路径（核心业务能力）
  - ⚠️ 边界 / 异常 / 兜底路径（易出问题，需重点验证）

## 一、对外消息入口与应用启动

- [ ] **UC-01** ✅ 接收一条会话消息并触发处理：外部渠道向系统发送一条带会话标识、可选文本和可选操作数据的消息 → 系统即时返回一条回复（卡片或纯文本），供渠道继续展示给用户。
- [ ] **UC-02** ✅ 消息按所属会话归类处理：消息带有会话标识（对应真实环境中一条消息串/根消息） → 后续处理都关联到同一场对话，不会与其他会话混淆。
- [ ] **UC-03** ⚠️ 仅携带操作数据、不带文本也可提交：消息文本可以缺省，只提供操作数据（如用户点击卡片按钮） → 系统仍能正常接收并返回处理结果，不因缺少文本而报错。
- [ ] **UC-04** ⚠️ 无法识别的消息内容：当消息内容匹配不到任何可用命令（如空文本或不认识的指令）时 → 系统不报错，而是返回统一的引导提示，告诉用户可尝试的命令（如"diagnose / 诊断"），方便重新输入。
- [ ] **UC-05** ✅ 服务启动后即可对外接收消息：启动完成后，消息入口即可接收外部消息并正常应答。
- [ ] **UC-06** ✅ 服务监听端口可配置：按配置指定端口对外提供服务（未配置则用默认端口 3000），便于不同环境部署。
- [ ] **UC-07** ✅ 启动成功有明确提示：服务就绪后，运维可看到服务已可访问的地址提示，便于确认启动状态。

## 二、流程编排与消息路由主干

- [ ] **UC-08** ✅ 新会话按关键词识别并开始向导：用户在尚无会话（或会话已关闭）的对话中发送文本，且命中某个已上线流程的关键词 → 系统为该对话新建会话（记录所用流程与语言），并展示该流程第一个应显示的步骤卡片（若首个步骤被跳过条件命中则展示下一个应显示步骤；若所有步骤都不适用则直接进入"请描述问题"提示）。
- [ ] **UC-09** ⚠️ 新会话文本未命中任何关键词：用户在无会话/会话已关闭的对话中发送文本，但未命中任何已上线流程 → 系统回复"未知指令，请尝试：diagnose / 诊断"提示，不创建会话。
- [ ] **UC-10** ✅ 卡片操作记录填写值并推进向导：用户在向导进行中的会话里点击/提交卡片上的某个选项（携带该步骤取值） → 系统记录该步骤的用户选择，将会话推进到下一步，并展示下一步卡片。
- [ ] **UC-11** ✅ 向导所有步骤填写完毕后转入自由对话：用户提交后向导步骤已全部走完 → 系统回复"准备就绪，请描述问题"（按会话语言显示中/英文），会话进入可自由输入问题描述的状态。
- [ ] **UC-12** ✅ 向导完成后自由文本触发处理并返回结果：用户完成向导后在同一对话中发送自由文本描述问题 → 系统将向导收集的信息连同用户描述一并提交处理，并把处理结果回复给用户。
- [ ] **UC-13** ✅ 向导进行中途误发自由文本会重新展示当前步骤：用户在向导尚未走完时发送普通文本而非卡片操作 → 系统不推进流程，只重新展示当前应处理的步骤卡片，等待用户通过卡片继续操作。
- [ ] **UC-14** ✅ 步骤按条件动态跳过：向导展示每一步前会先判断该步骤在当前已收集信息下是否需要展示 → 不满足条件的步骤自动跳过，用户只看到与自己情况相关的步骤，会话进度同步更新到实际展示的步骤位置。
- [ ] **UC-15** ✅ 全部步骤跳过后直接进入自由对话：从当前位置逐步判断展示条件，若一路跳到结尾都没有需要展示的步骤 → 系统跳过卡片环节，直接回复"准备就绪，请描述问题"提示。

## 三、命令解析与触发词匹配（配置驱动）

- [ ] **UC-16** ✅ 英文触发词命中启动对应功能：消息中包含某功能配置的任意英文触发词（不区分大小写、出现在任意位置即可，如 "please diagnose this"） → 系统识别为该功能请求，并以英文界面开始该功能引导流程。
- [ ] **UC-17** ✅ 中文触发词命中启动对应功能：消息中包含某功能配置的任意中文触发词（原文包含即可，如 "帮我做个诊断"） → 系统识别为该功能请求，并以中文界面开始该功能引导流程。
- [ ] **UC-18** ✅ 英文触发词优先于中文触发词：同一条消息可能同时命中同功能的英文与中文触发词时，系统按英文触发词判定语言 → 该功能以英文界面开始。
- [ ] **UC-19** ⚠️ 未命中任何已上线功能的触发词：消息不包含任何已上线功能的触发词 → 系统不启动任何引导流程（判定为非功能请求，交由默认处理）。
- [ ] **UC-20** ✅ 多功能按上线顺序匹配：同时上线多个功能时，按功能上线的先后顺序逐个检查是否命中，命中即采用该功能，不再检查后续功能。
- [ ] **UC-21** ✅ 按功能标识精确取用配置：已知某功能标识（如 "diagnose"）时，可直接取出该功能的完整配置（触发词、结束方式、步骤定义、是否收集反馈等）。
- [ ] **UC-22** ⚠️ 请求不存在的功能标识：传入的功能标识在已上线功能中不存在 → 系统返回"未找到该功能"，不启动任何流程。
- [ ] **UC-23** ✅ 新增功能无需改动通用流程：新增一份功能配置（含触发词、步骤等定义）即可上线一个新的可触发功能，无需改动其余通用处理逻辑。
- [ ] **UC-24** ✅ 「诊断」功能触发词与引导：英文触发词 "diagnose"、中文触发词 "诊断"；命中后进入两步引导：先选择环境（QAT / UAT / PROD 三选一），再输入分支（默认值 "main"，可采用默认或自行输入）。
- [ ] **UC-25** ✅ 「诊断」功能结束方式与反馈：该功能采用对话式结束方式（允许多轮交互后自然收尾），且未开启功能结束后的用户反馈收集。

## 四、步骤引擎与卡片渲染（配置驱动）

- [ ] **UC-26** ✅ 步骤默认展示：某步骤未配置显示条件时 → 始终展示给用户，不会被跳过。
- [ ] **UC-27** ✅ 按条件展示步骤：某步骤配置了"仅当此前收集的答案等于某值才展示"，且用户此前回答确实等于该值 → 该步骤会被展示。
- [ ] **UC-28** ⚠️ 按条件隐藏步骤：同上条件存在，但用户此前回答不等于设定值（或该字段尚未收集到任何值） → 该步骤被跳过、不展示。
- [ ] **UC-29** ✅ 单选题卡片：步骤为"单选"时 → 展示为一组可点击按钮，按钮文案按用户当前语言（中/英）显示。
- [ ] **UC-30** ✅ 多选题卡片：步骤为"多选"时 → 展示为一组可点击按钮（与单选共用同一展示形式）。
- [ ] **UC-31** ⚠️ 选项为空兜底：单选/多选步骤未提供任何选项时 → 展示空按钮区域，不报错、不崩溃。
- [ ] **UC-32** ✅ 文本输入题卡片：步骤为"输入"时 → 展示为可填写的输入表单，并带"提交"按钮。
- [ ] **UC-33** ✅ 提交按钮双语：输入步骤的提交按钮文案随当前语言显示为"提交"或"Submit"。
- [ ] **UC-34** ✅ 输入框默认值提示：输入步骤配置了默认值时 → 输入框占位提示显示该默认值（如"默认：xxx" / "default: xxx"）。
- [ ] **UC-35** ✅ 输入框空态提示：输入步骤未配置默认值时 → 输入框占位提示显示通用引导文案（"请输入…" / "Type here…"），按当前语言展示。
- [ ] **UC-36** ✅ 卡片标题双语：每张步骤卡片标题按用户当前语言（中/英）展示对应文案。
- [ ] **UC-37** ✅ 卡片辅助说明：步骤配置了辅助说明时 → 卡片顶部额外展示一行说明文字，帮助用户理解该步骤。
- [ ] **UC-38** ⚠️ 无辅助说明时不展示：步骤未配置辅助说明（或当前语言无对应文案）时 → 卡片不显示该行，不会出现空白或占位符。
- [ ] **UC-39** ✅ 卡片视觉统一：所有步骤卡片使用统一强调色（青绿色）标题栏，保持视觉一致性。
- [ ] **UC-40** ✅ 会话归属标识：用户在卡片上的操作会被正确归属回对应会话与所属场景（如"配置向导"或"反馈流程"），不会串场景。
- [ ] **UC-41** ✅ 模板变量替换：文案中包含"{{ 字段名 }}"占位符时 → 展示时替换为已收集数据中对应字段的实际值。
- [ ] **UC-42** ✅ 多级字段引用：占位符支持引用多级字段（如"collected.env"这类点号路径） → 正确取到嵌套数据中的对应值并替换。
- [ ] **UC-43** ⚠️ 未知变量兜底：占位符引用的字段在已收集数据中不存在（或路径中间某级不存在）时 → 该占位符按空字符串处理，不报错、不残留占位符原文。
- [ ] **UC-44** ⚠️ 空值变量兜底：占位符引用的字段值为空时 → 同样按空字符串处理展示。

## 五、会话状态机与线程隔离生命周期

- [ ] **UC-45** ✅ 按会话线程建立独立会话：某线程首次触发指令时，系统为其建立专属会话，初始状态为"进行中"、步骤从头开始、已收集信息为空 → 后续该线程所有交互都在这个专属会话里推进，不与其他线程互相干扰。
- [ ] **UC-46** ✅ 同一线程重复触发不会重开会话：线程已有会话时再次触发指令（无论哪条指令） → 系统直接复用已有会话（保留当前状态、已走步骤、已收集信息，以及最初记录的指令与语言），不重置进度、不切换指令、不产生第二个会话。
- [ ] **UC-47** ✅ 查询线程当前会话：按线程标识查询 → 返回该线程当前会话记录（状态、步骤、已收集信息等）；从未建立过会话则查无结果。
- [ ] **UC-48** ✅ 会话进度持续保存：会话推进过程中（进入下一步、收集新信息、状态变化） → 最新内容会被保存，下次查询/交互看到的是最新进度而非初始状态。
- [ ] **UC-49** ✅ 主动关闭会话：某线程会话被显式关闭 → 状态永久变为"已关闭"（终态），之后不再被当作进行中的会话处理。
- [ ] **UC-50** ⚠️ 关闭一个不存在的线程会话：对没有会话记录的线程执行关闭 → 不报错、无任何影响（静默忽略）。
- [ ] **UC-51** ✅ 会话闲置超时自动关闭：一个"进行中"会话距上次更新达到或超过设定闲置时长 → 该会话被自动收尾为终态，并作为本轮结果返回，避免无人回应的会话无限期挂起。
- [ ] **UC-52** ⚠️ 闲置检查不影响未超时或非进行中的会话：会话仍在闲置时限内，或已处于"等待反馈"/"已关闭"等非"进行中"状态 → 不会被自动关闭，维持原状态。
- [ ] **UC-53** ✅ 系统按固定节奏自动巡检闲置会话：系统运行期间每隔约 5 分钟自动巡检一遍所有会话，把闲置超过约 2 小时的"进行中"会话收尾 → 用户长时间未响应的会话无需人工干预即自动收尾，不会无限占用资源。
- [ ] **UC-54** ⚠️ 系统停止服务时自动巡检随之停止：服务关闭/重启时 → 定时巡检随之停止，不会在服务下线后继续残留运行。
- [ ] **UC-55** ✅ 会话具备完整生命周期状态：一个线程的会话可处于"进行中"、"等待反馈"、"已关闭"三种状态之一（新建/复用时为"进行中"，闲置超时或主动关闭后变为"已关闭"终态，不可逆转），反映该线程当前所处的业务交互阶段。

## 六、诊断处理与结果回传

- [ ] **UC-56** ✅ 接收诊断请求并生成报告：传入包含功能名、语种、已收集信息（如环境/分支）与用户问题描述的一次诊断请求 → 系统返回一段包含环境、分支、问题原文与根因结论（附置信度）的诊断报告。
- [ ] **UC-57** ✅ 按语种输出对应语言的报告：语种为英文 → 报告标题、字段标签与结论均为英文；语种为中文 → 返回中文报告，内容结构一致（环境/分支/问题/根因/置信度）。
- [ ] **UC-58** ✅ 报告中回填已收集的上下文信息：已收集信息提供了具体环境值和分支值 → 报告原样展示这两个值，并原样引用用户提交的问题原文。
- [ ] **UC-59** ⚠️ 缺失上下文信息时的兜底展示：已收集信息未提供环境或分支时 → 报告对应字段展示为占位符"?"，不报错、不中断报告生成。
- [ ] **UC-60** ✅ 诊断存在处理耗时：每次请求处理有短暂等待后才返回结果，模拟真实诊断所需的处理时间（响应有可感知但很短的延迟，而非瞬时返回）。
- [ ] **UC-61** ✅ 诊断结果交由上层回传：诊断环节只产出报告文本，不直接发送到用户对话渠道 → 结果由上层流程负责回传给用户所在的会话线程。

## 验证重点（最易出问题的边界用例）

以下用例是回归 / 迁移后最需要重点核对的边界，QA 建议优先覆盖：

1. **无法识别的输入引导**（UC-04 / UC-09 / UC-19）：空文本、乱输入、未命中触发词时，是否稳定返回统一引导提示，且**不创建会话**。
2. **触发词语言判定**（UC-16~UC-18）：同一句话同时含中英文触发词时，是否按"英文优先"判定语言，界面语言是否正确。
3. **向导中途误操作**（UC-13）：向导未走完就发自由文本，是否只是重新展示当前步骤而**不推进、不误判为问题描述**。
4. **步骤条件跳过与全跳过**（UC-14 / UC-15 / UC-27 / UC-28）：按已收集答案跳过步骤是否正确，全部跳过时是否直接进入"请描述问题"。
5. **仅操作数据无文本**（UC-03）：点击卡片按钮（无文本）时能否正常处理、不报错。
6. **模板变量兜底**（UC-43 / UC-44）：未知字段/空值占位符是否按空字符串处理，绝不把"{{ xxx }}"原文暴露给用户。
7. **选项为空兜底**（UC-31）：配置缺选项时卡片是否不崩溃。
8. **会话复用不重开**（UC-46）：同一线程重复触发是否绝不重置进度、不切换指令、不产生第二个会话。
9. **闲置自动收尾**（UC-51~UC-54）：约 2 小时闲置的"进行中"会话是否被自动收尾，且**不误伤**未超时或"等待反馈"/"已关闭"会话；服务重启后巡检是否不残留。
10. **诊断缺上下文兜底**（UC-59）：缺环境/分支时报告是否用"?"占位而非报错或中断。
