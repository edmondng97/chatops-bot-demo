## 三、不一致（INCONSISTENT）+ 功能性分析

### UC-51 — 会话闲置超时自动关闭

- 证据：`backend/src/modules/session/session.service.ts:44-53` 的 `sweepIdle()` 只把状态改为 `CLOSED` 并返回被关闭的线程标识列表；`backend/src/modules/session/session-sweeper.service.ts:14-16` 定时调用时直接丢弃返回值，未把结果作为消息回传给对应对话。
- 问题：闲置会话在后台被静默关闭，但没有任何通知链路把这一状态变化告知用户。
- 业务影响：闲置太久没人搭理的用户对话会在后台被静默标记结束，但用户不会收到"会话已自动结束"的提示，可能误以为还能继续之前的操作，造成后续消息发出去却得不到预期响应的困惑。
- 严重度：MEDIUM

### UC-04 — 无法识别的消息内容返回统一引导

- 证据：`backend/src/modules/flow/flow-orchestrator.service.ts:39,46`，写死英文文案 "Unknown command. Try: diagnose / 诊断"，未按 `existing?.locale` 区分中英文，且未见明确排除报错的兜底测试。
- 问题：引导提示文案固定为中英混排，未按用户语言环境做区分展示。
- 业务影响：中文用户收到的引导提示是中英混排的固定文案，而不是像会话关闭提示那样按用户语言展示，体验不完全一致，但功能本身不报错、仍能给出引导，不影响用户继续操作。
- 严重度：LOW

### UC-22 — 请求不存在的功能标识

- 证据：`flow-registry.service.ts:13-15` 的 `get()` 方法在找不到对应功能时返回 `undefined`，未返回"未找到该功能"类的明确提示；调用方 `flow-orchestrator.service.ts:45` 后续处理在提供的代码片段中未显示对 `undefined` 的兜底文案。
- 问题：功能查找失败时返回值缺少明确的错误语义，兜底提示文案链路未在本次核对范围内完全确认。
- 业务影响：若历史会话记录的功能标识失效，用户可能得到通用/静默失败而非清晰的"未找到该功能"提示（需进一步核实 orchestrator 后续分支，此域文件本身只提供查找能力，不产出提示文案）。
- 严重度：LOW
